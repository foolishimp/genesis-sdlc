# GAP: genesis_sdlc's Engine Runs Against Its Own Build Source — GCC 1.0 Must Not Execute GCC 2.0

**Author**: Claude Code
**Date**: 2026-03-20T12:00:00Z
**Addresses**: Self-hosting boundary, three-layer install architecture (ADR-005), bootstrap model
**For**: all

## Summary

genesis_sdlc's running engine (GCC 1.0, installed at `.genesis/`) currently reads its Package from `gtl_spec/packages/genesis_sdlc.py` — which is a live-editable near-duplicate of the build source at `builds/python/src/genesis_sdlc/sdlc_graph.py`. This means the compiler being built (GCC 2.0) is also the compiler being run. This is a bootstrap boundary violation.

## The Problem

The genesis product is the pair: abiogenesis (engine) + genesis_sdlc (standard library). The bootstrap model is:

```
GCC 1.0 (released abg + gsdlc, installed at .genesis/)  →  builds GCC 2.0 (next version, in builds/)
```

The installed engine at `.genesis/` is GCC 1.0. It should evaluate the workspace against the **released** Package — also GCC 1.0. But `genesis.yml` points at `gtl_spec.packages.genesis_sdlc:package`, which is the **in-progress build source** (GCC 2.0). So:

```
CURRENT (wrong):
  .genesis/ engine (GCC 1.0)  reads  gtl_spec/ (GCC 2.0 being edited)

  Editing the spec changes what the running engine evaluates against.
  The compiler you're building is rewriting the compiler that's running.

CORRECT:
  .genesis/ engine (GCC 1.0)  reads  .genesis/spec/ or .genesis/workflows/ (GCC 1.0 Package)

  The running engine is stable. The build source is just source code.
  Only the sandbox test runs GCC 2.0.
```

## Why This Keeps Coming Up

This confusion recurs because genesis_sdlc is simultaneously:
1. **A product being built** — `builds/python/src/genesis_sdlc/` is the source
2. **A downstream project using itself** — `.genesis/` is the installed compiler
3. **Its own test subject** — `gtl_spec/` looks like the project's local Package customisation

For any other project (e.g. `genesis_data_mapper`), the separation is clean:
- `.genesis/` has the installed engine + released Package
- `gtl_spec/packages/genesis_data_mapper.py` is the project's own spec (Layer 3)
- Nobody confuses them because the project doesn't also ship the Package

For genesis_sdlc, `gtl_spec/packages/genesis_sdlc.py` is both "the project's local Package" AND "a near-copy of the build source." Two files that should say the same thing, in two locations, maintained by different processes. They drift.

## The Rule

**The running engine must evaluate against the released Package, not the build source.**

- `genesis.yml` should resolve to the installed Package in `.genesis/` (GCC 1.0)
- `builds/python/src/genesis_sdlc/sdlc_graph.py` is source code being developed (GCC 2.0)
- `gtl_spec/packages/genesis_sdlc.py` should not exist in this repo, or should be generated and not hand-edited
- The **only** place GCC 2.0 runs as a compiler is the sandbox e2e test (`unit_tests→integration_tests` edge)

## What This Changes

| Current | Proposed |
|---------|----------|
| `genesis.yml` → `gtl_spec.packages.genesis_sdlc:package` | `genesis.yml` → installed Package in `.genesis/` |
| `gtl_spec/packages/genesis_sdlc.py` is live-edited | Either removed or generated by installer (not hand-edited) |
| Editing spec changes running engine behaviour | Editing spec changes only the build source; engine is stable |
| Spec drift between `builds/` and `gtl_spec/` is possible | Single source of truth in `builds/` |

## The GCC Test

> Can I delete `builds/` and still have a working, stable development environment?

**Yes** — `.genesis/` has everything the engine needs. The build source is what's *being developed*, not what's *running*.

> Can I delete `.genesis/` and still develop?

**No** — that's the installed compiler. Reinstall from the released version.

> Can I edit `builds/python/src/genesis_sdlc/sdlc_graph.py` without affecting my current `gen-gaps` output?

**Currently no** (because `gtl_spec/` drifts or is a copy). **Should be yes.**

## Recommended Action

1. Change `genesis.yml` to point at the installed Package in `.genesis/` rather than `gtl_spec/`
2. Remove or auto-generate `gtl_spec/packages/genesis_sdlc.py` in this repo
3. Document this as the self-hosting boundary rule: the engine runs GCC 1.0, the sandbox tests GCC 2.0
4. This is a V1 close-out item — fix before the self-hosting gate
