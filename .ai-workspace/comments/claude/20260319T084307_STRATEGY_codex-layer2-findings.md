**Raised by**: Claude Code (responding to Codex review)
**Status**: Pending human decision

## The Codex Findings (Summary)

Codex identified three bugs in the v0.2.0 three-layer install architecture (REQ-F-BOOT-003/004/005):

1. **Layer 2 is never used at runtime.** `install_immutable_spec()` copies `sdlc_graph.py` into `.genesis/spec/genesis_sdlc.py`, but `genesis.yml` still points to `gtl_spec.packages.<slug>:package` (Layer 3). Layer 2 is dead code ‚Äî it cannot affect engine evaluation.

2. **Layer 2 and Layer 3 have inconsistent requirement registries.** The file copied to Layer 2 (`sdlc_graph.py`) had a shorter requirements list than Layer 3 (`genesis_sdlc.py`). This means two assets claiming to represent the same release would produce different `spec_hash` values. (This specific inconsistency is now fixed ‚Äî `sdlc_graph.py` was updated during INT-003 implementation to carry the full requirements list.)

3. **Import namespace is wrong.** The ADR says downstream projects import `genesis.spec.genesis_sdlc`, but `.genesis/spec/__init__.py` makes the import path `spec.genesis_sdlc`. `genesis.spec.genesis_sdlc` raises `ModuleNotFoundError`. (This only matters if Layer 3 is a thin wrapper ‚Äî which it currently is not.)

---

## Root Cause

The three-layer architecture was added in v0.2.0 with the intent that Layer 3 (local spec) would be a **thin wrapper** that imports from Layer 2 (immutable released spec):

```python
# gtl_spec/packages/{slug}.py (thin wrapper intent)
from spec.genesis_sdlc import package, worker
# project-specific overrides here
```

But the implementation generates a **full copy** of the genesis_sdlc spec as Layer 3, not a thin wrapper. So Layer 2 has no importer and is never reached. The architecture described in ADR-005 was not actually implemented.

---

## Two Viable Paths Forward

### Option A: Remove Layer 2 (simplest, honest)

Remove `install_immutable_spec()` from `install.py`. Remove the verification of `.genesis/spec/genesis_sdlc.py` from `_verify()`. Update or remove REQ-F-BOOT-003 (change to: "installer generates full local spec from template" ‚Äî which is what actually happens). REQ-F-BOOT-004 and REQ-F-BOOT-005 remain accurate.

**Pros**: No dead code. Tests reflect reality. No namespace confusion.
**Cons**: Loses the "immutable methodology baseline" promise ‚Äî reinstalling genesis_sdlc can't update the methodology in dependent projects unless they still have the stub marker (which is one-time-only).

### Option B: Implement Thin Wrapper (correct three-layer intent)

Change `install_sdlc_starter_spec()` to generate a thin wrapper instead of a full copy:

```python
# gtl_spec/packages/{slug}.py ‚Äî generated by gen-install
# Implements: local customizations for {slug}
# To customize: modify package below; reinstalling genesis_sdlc updates spec.genesis_sdlc.
from spec.genesis_sdlc import package as _base, worker

# Inherit the base SDLC graph. Add project-specific REQ keys and assets here.
package = _base
```

The stub check becomes: does Layer 3 import from `spec.genesis_sdlc`? If yes, it's still using the base ‚Äî safe to update on reinstall. If user has customized it (different import or no import), leave it alone.

The `_verify()` check for Layer 2 stays. Layer 2 (`spec.genesis_sdlc`) is now actually used at runtime.

Fix the namespace in ADR-005: `spec.genesis_sdlc` not `genesis.spec.genesis_sdlc`.

**Pros**: Three-layer promise is real. Dependent projects get methodology updates on reinstall without losing customizations. REQ-F-BOOT-003/004/005 are correctly implemented.
**Cons**: More complex. Requires changing the starter spec template and stub detection logic.

---

## My Recommendation

**Option B** ‚Äî implement the thin wrapper. The three-layer architecture is genuinely useful for dependent projects. The fix is bounded:

1. Change `install_sdlc_starter_spec()` to write a thin-wrapper template (not a full copy)
2. Change stub detection: look for `from spec.genesis_sdlc import` (instead of `"spec‚Üíoutput"`)
3. Fix ADR-005 namespace: `spec.genesis_sdlc` (not `genesis.spec.genesis_sdlc`)
4. Add a test: `test_starter_spec_imports_layer2` ‚Äî verifies Layer 3 imports from `spec.genesis_sdlc`
5. Add a test: `test_layer2_importable` ‚Äî verifies `spec.genesis_sdlc` resolves after install

This is a MINOR release change (no new REQ keys; REQ-F-BOOT-003/004/005 already exist and are correctly named; the implementation just wasn't wired up).

For genesis_sdlc itself (dogfooding its own spec), the local spec at `gtl_spec/packages/genesis_sdlc.py` stays as a full spec ‚Äî it IS the source, not a wrapper. Only dependent projects get thin wrappers.
