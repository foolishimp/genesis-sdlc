## The Dogfooding Principle (do not lose this)

genesis_sdlc enforces version-sandboxed testing on downstream projects. It must enforce the same on itself. The principle is:

> Every test layer runs against an explicitly pinned version of the code. Unit/integration tests pin to `builds/python/src/` (RC source). UAT tests pin to a fresh sandbox install (RC packaged). No test layer inherits its import path from the environment.

REQ-F-TEST-003 is NOT a PYTHONPATH fix. It is the mechanical expression of version-sandboxing applied to the development loop. The UAT edge already does this correctly (sandbox install). The unit/integration evaluator commands do not — they rely on environment inheritance. REQ-F-TEST-003 closes that gap.

---

## Three-Layer Architecture (do not lose this)

When genesis_sdlc v0.2.0 is installed into any project:

```
.genesis/genesis/          ← abiogenesis runner (immutable, already correct)
.genesis/gtl/              ← GTL primitives (immutable, already correct)
.genesis/spec/             ← released genesis_sdlc methodology (NEW — REQ-F-BOOT-003)
gtl_spec/packages/         ← local project spec (mutable, project-owned)
```

The `genesis_sdlc` self-hosting special case: during development, `gtl_spec/packages/genesis_sdlc.py` IS the thing being built. Both copies coexist:
- `.genesis/spec/genesis_sdlc.py` — frozen v0.1.6 (prior release)
- `gtl_spec/packages/genesis_sdlc.py` — v0.2.0 under development

After release they are identical until the next development cycle begins.
