# ABG 1.0 Handoff: Small Real-FP Set and Release-Evidence Next Steps

## Purpose

This note summarizes the current architectural position and recommends the next steps for Claude.

The goal is no longer broad cleanup in the abstract. The goal is to finish ABG to the point where release can be judged from honest evidence.

## Current Position

### 1. ABG identity is now clearer

The working model is:

- `abg` is kernel SDK + kernel app
- `abiogenesis/` is source, not an installed self-hosting project
- self-hosting, if desired later, must be an explicit project/package proof
- `genesis_core` should be treated as a reference/bootstrap package, not silently conflated with a universal domain default

This removed a large amount of confusion from prior work.

### 2. The archive/test infrastructure is now materially better

Recent progress means ABG now has:

- persistent archived scenario runs
- manifest persistence
- result persistence
- event persistence
- replay summary aggregation for nested sandboxes
- app-level `assess-result` result ingestion

That means the system is now close to producing release evidence rather than just test results.

### 3. The current scenario suite proves protocol better than semantics

The disjoint scenario suite is now good at proving:

- install/bind behavior
- F_D gating
- F_P dispatch
- manifest truth
- result ingestion
- event/postmortem archive quality

But it is still weak at proving:

- real F_P semantic correctness
- realistic iterative repair behavior
- realistic shipping-system UAT

The present fixture pattern still tends to simulate a passing F_P result rather than truly judge the produced artifact against the hop contract.

## Key Insight

Do **not** expand the matrix broadly right now.

Do **not** chase many more synthetic edge types yet.

The right move is:

- keep a **small targeted set**
- make those scenarios **real**
- make them **iterative**
- use them as **release evidence**

This is more valuable than a large shallow matrix.

## Recommended Core Set

Promote these three scenarios into the primary ABG 1.0 release-evidence suite.

### 1. `intent -> requirements`

Why keep it:

- cleanest proof of broad intent becoming structured normative output
- tests whether the system can carry process and intent without hiding behind codegen

What must become real:

- deterministic F_P judge verifies all expected REQ keys exist
- checks scope coverage
- rejects implementation leakage
- rejects under-specified or malformed output

Recommended iterative shape:

- pass 1: incomplete coverage
- pass 2: implementation leakage or poor specificity
- pass 3: acceptable converged artifact

This should become the canonical “iterative normative refinement” scenario.

### 2. `design -> data_schema`

Why keep it:

- best current technical-context scenario
- strongest test that F_P uses context correctly rather than just writing plausible text

What must become real:

- deterministic judge checks required entities
- naming conventions
- integrity constraints
- migration-safety markers

Recommended iterative shape:

- pass 1: missing constraints
- pass 2: wrong naming / weak schema contract
- pass 3: converged schema

This should become the canonical “iterative technical refinement” scenario.

### 3. `requirements -> uat_tests`

Why keep it:

- closest current bridge from kernel behavior to shipping-system behavior
- strongest candidate for release-evidence scenario

What must become real:

- deterministic judge checks every REQ is covered
- steps are executable
- expected results are observable
- edge cases are included

Recommended iterative shape:

- pass 1: partial requirement coverage
- pass 2: vague steps / missing expected results
- pass 3: converged UAT set

This should become the canonical “acceptance artifact” scenario.

## What to Deprioritize

### `brief -> article`

Keep as a demonstration/usefulness scenario, but do not make it a release-critical ABG 1.0 proof.

It is lower signal than the three core scenarios above.

### `modules × design -> code`

Keep, but do not treat it as the next critical scenario.

It is valuable, but it is also the most likely to sprawl. Bring it back only after the three core scenarios above are honest and stable.

### `kernel_safety`

Keep as infrastructure/property coverage.

Do not confuse it with release-evidence F_P behavior.

## Real F_P Standard

For the core scenarios above, “real F_P” should mean:

1. `iterate` produces `fp_manifest_path`
2. the fixture-side F_P runner reads the manifest
3. the runner inspects the produced artifact
4. the runner writes a result JSON to `result_path`
5. `assess-result` ingests that JSON
6. convergence only happens if the artifact actually satisfies the scenario-specific rules

This is the difference between:

- protocol simulation

and:

- real judged scenario behavior

## Iterative Scenario Standard

Each real-F_P core scenario should become a **layered scenario** with multiple F_P turns in the same archived run.

The recommended form is:

- initial unconverged state
- first F_P attempt fails for one meaningful reason
- second F_P attempt improves but still fails
- final F_P attempt converges

This is more revealing than a single-shot green because it shows:

- whether the manifest supports repair
- whether event/postmortem traces remain useful across iterations
- whether the destination/work surface remains consistent across turns

## Release-Evidence Standard

Before cutting ABG 1.0, the evidence should come from:

### A. kernel/app contract proof

- install truth
- bind truth
- manifest truth
- result ingestion truth
- event/postmortem truth

### B. three real-F_P iterative scenarios

- `intent -> requirements`
- `design -> data_schema`
- `requirements -> uat_tests`

### C. postmortem review of archived runs

Each archived run should be good enough to answer:

- what happened
- why it failed
- why it converged
- what context was used
- what artifact was produced
- what result was ingested

Release approval should be based on those archives, not just pytest green.

## Remaining Important Gaps

Claude should keep these in mind while doing the next round:

### 1. `workflow_version` in scenarios

Scenario archives still tend to show `workflow_version: unknown` because the sandboxes do not carry active workflow metadata.

This matters for release-grade provenance.

### 2. Semantic correctness is still weaker than protocol correctness

The archive system is close.

The semantic judges are not yet strong enough to certify the work products honestly.

### 3. Some stale self-hosting tests still remain

These are not part of the real-F_P release-evidence path and should not be allowed to distract from it.

## Concrete Next Steps for Claude

### Step 1

Finish the provenance gap for scenario archives:

- make scenario sandboxes capable of carrying known workflow metadata where appropriate
- ensure assessed events can carry a known `workflow_version`

### Step 2

Upgrade the three core scenarios to real judged F_P behavior:

- `intent -> requirements`
- `design -> data_schema`
- `requirements -> uat_tests`

Replace synthetic “pass” assumptions with deterministic artifact judges.

### Step 3

Add layered multi-iteration behavior to those three scenarios.

The scenario should archive:

- first failed F_P attempt
- second improved attempt
- final converged attempt

### Step 4

Keep postmortem quality as a first-class acceptance criterion.

Do not only ask:

- did it pass?

Also ask:

- would a release reviewer understand what happened from the archive alone?

### Step 5

Only after those three scenarios are honest and stable:

- revisit `modules × design -> code`
- revisit additional edge classes
- return focus to `gsdlc`

## Final Recommendation

ABG is close enough that the next work should be disciplined and narrow.

Do not broaden the scenario set yet.

Do not over-optimize synthetic coverage.

Instead:

- finish provenance
- make a small core set real
- make it iterative
- use archived postmortems as release evidence

That is the shortest path to an honest ABG 1.0 decision.
