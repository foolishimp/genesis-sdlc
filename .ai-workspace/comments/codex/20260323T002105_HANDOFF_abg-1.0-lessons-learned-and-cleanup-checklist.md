# HANDOFF: ABG 1.0 Lessons Learned And Cleanup Checklist

**Date**: 2026-03-23T00:21:05+11:00
**Context**: post-release lessons from ABG 1.0 stabilization, to be applied to GSDLC cleanup

## Summary

ABG reached 1.0 not because every edge case was polished, but because the last major architectural lie was exposed and removed:

- the kernel install had to become structurally honest
- immutable install territory had to stop shipping project/domain spec truth
- mutable runtime state had to move under `.ai-workspace`
- live F_P protocol had to become real
- clean install had to become a release gate, not a postscript

The clean install into `test_install` was the decisive test. It exposed contamination that the source repo and self-referential test environment had been hiding.

## Core Lessons

### 1. Self-referential repos hide boundary bugs

When the source repo is also treated as the project, broken defaults still appear to work because the expected repo paths already exist.

Examples ABG exposed:
- installed default package pointing at `builds/claude_code/...`
- hardcoded `.gsdlc/release/genesis.yml` in kernel discovery
- installed spec namespace (`.genesis/gtl_spec`) masquerading as kernel truth

Rule:
- never trust the source repo as proof of install correctness

### 2. Clean install is stronger than a green suite

The suite can validate protocol and behavior, but only a clean install answers:

- what must exist for the kernel to function?
- what gets shipped into immutable territory?
- which assumptions are accidentally satisfied only by the source tree?

Rule:
- every release must include a clean install into an empty target

### 3. Kernel, domain, and project must be separate territories

ABG only became honest when these roles were separated:

- `.genesis/` = immutable kernel install
- `.ai-workspace/` = mutable runtime state/history/evidence
- domain contract = explicit runtime contract outside kernel internals
- project/package = explicit binding, not implied by the kernel

Rule:
- if a file contains project truth, it does not belong in vanilla `.genesis/`

### 4. Fallbacks become architecture if left in place

Legacy pathing and bootstrap fallbacks are dangerous because tests and docs slowly normalize them.

Examples:
- `.genesis/active-workflow.json`
- hardcoded `.gsdlc/release/genesis.yml`
- default `genesis_core` binding

Rule:
- fallback paths must stay visibly fallback, or they become de facto standards

### 5. Integration tests can be truer than the installer

The ABG integration tests had already proven the right operating model:

1. install kernel
2. write explicit project package
3. bind config explicitly
4. run the hop

The installer was lagging behind the test-proven architecture.

Rule:
- when tests and installer disagree, do not assume the installer is more correct

### 6. Live qualification and clean install validate different truths

Two different release questions emerged:

- clean install: is the architecture honest?
- live qualification: is the dispatch/prompt/worker behavior sufficient?

Both were necessary.

Rule:
- do not substitute one for the other

### 7. Real protocol matters more than simulated pass events

The early harness faked convergence by emitting `assessed` directly. That was not good enough.

The real improvement was:
- `iterate` writes manifest
- result file is written to `result_path`
- `assess-result` ingests it
- event stream records the outcome

Rule:
- tests may simulate the use case, but they must not bypass the declared protocol

### 8. WorkingSurface / typed-contract drift is release debt

A key near-MVP lesson was that the GTL type contract and app behavior can diverge quietly.

If `iterate()` claims to produce a `WorkingSurface`, but real artifact evidence lives elsewhere, that drift will become expensive after release.

Rule:
- where the type contract is authoritative, the app must carry the same evidence surface

### 9. Archive quality matters

Postmortem archives became valuable only when they were:
- persistent
- honest
- complete enough to explain failure and convergence

This surfaced its own checklist:
- no blind summaries
- no unlogged subprocesses
- manifest/result protocol must be real
- per-iteration artifacts must not overwrite each other

Rule:
- release evidence is not just pytest output; it is the archived explanation of what happened and why

### 10. Source comments, examples, and docs can reintroduce old architecture

Even after install correctness improved, stale comments/examples still pointed at:
- `gtl_spec.packages.genesis_core`
- self-hosting fallbacks
- old module counts

Rule:
- stale examples are not harmless in boundary-sensitive systems; they teach the wrong architecture

## Release Checklist Template

Use this for GSDLC cleanup and future releases.

### A. Clean Install Audit

- install into an empty target
- inspect full installed tree
- confirm immutable vs mutable territory split
- confirm no authored-source or project-spec leakage into kernel territory
- confirm no hardcoded repo-local paths survive

### B. Bootstrap Contract Audit

- bootstrap entrypoint is singular and clear
- no hardcoded domain-specific discovery path in kernel
- domain override is explicit via config, not guessed by the kernel
- no default package binding unless it is truly generic

### C. Package/Spec Audit

- installed default packages are either:
  - absent, or
  - truly generic
- no REQ-bearing project spec in vanilla kernel install
- no self-hosting assumptions disguised as defaults

### D. Mutable State Audit

- events under `.ai-workspace`
- workflow/runtime state under `.ai-workspace`
- manifests/results under `.ai-workspace`
- `.genesis` remains immutable after install

### E. Test Truth Audit

- integration tests use the real protocol
- no direct event bypasses where the app contract says otherwise
- live lane and deterministic judged lane are clearly separated
- archives are complete and reviewable

### F. Live Qualification Audit

- choose a small number of high-value hops
- require real agent execution using only the production dispatch surface
- judge outputs deterministically
- define success-rate thresholds
- archive manifest, result, artifact, events, and transcript/logs

### G. Documentation / Example Audit

- no examples that reference removed or fallback-only paths
- no stale self-hosting language if the product is kernel-first
- no stale module-count or package-binding claims
- release docs describe the actual install model

## Applying This To GSDLC

The same questions now apply to GSDLC:

- what belongs in domain install territory vs authored source?
- what is explicit runtime contract vs accidental repo residue?
- what does a clean `abg + gsdlc` install into an empty target really need?
- which parts of `.claude`, `.gsdlc/release`, and `.ai-workspace` are actual domain runtime, and which are legacy contamination?

## Final Takeaway

ABG 1.0 was stabilized by forcing one hard question:

> If I install this into an empty target, what truth does it ship by default?

That question should become permanent. It is the fastest way to surface hidden self-reference, leaked assumptions, and fallback paths that quietly became architecture.
