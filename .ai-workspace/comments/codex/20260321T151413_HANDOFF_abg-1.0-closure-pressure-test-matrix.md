**Author**: Codex
**Date**: 2026-03-21T15:14:13
**For**: Claude Code
**Purpose**: Incorporate into the ABG 1.0 strategy and implementation plan

## Position

The right way to close `abiogenesis` at 1.0 is not to assume the current model is complete, and not to keep extending it opportunistically. The right method is to pressure-test the model against plausible future feature sets and ask, for each one:

`Can this be expressed with current GTL + current abg primitives + composition law?`

If yes, it is not an ABG blocker.

If no, identify the exact missing abstraction and determine whether it is:
- a spec clarification
- a composition-law gap
- a build/runtime realization gap
- or a true missing primitive

This gives a disciplined way to close ABG 1.0 while still being honest that later evolution may be required.

## Proposed Two-Part Structure

### Part 1 — ABG 1.0 Closure

This part should establish:

1. **Kernel contract**
   `abg` owns:
   - single-hop execution
   - convergence and gate ordering
   - event/provenance substrate
   - context binding
   - governed gap handling
   - hop-local homeostasis

2. **Composition law**
   Higher-order workflows are expressed without new primitives where possible:
   - `ObserverModel = composed context surface`
   - `Gap = delta(observed_state, observer_model)`
   - `Intent -> CompositionSet -> routed workflow(s) -> observed result -> new/refined intent`

3. **ABG 1.0 closure decision**
   Declare what is in 1.0, what is explicitly deferred, and what would force post-1.0 evolution.

### Part 2 — Review GSDLC Against ABG 1.0

Only after Part 1 is fixed:
- evaluate `gsdlc` as a build/service realization over the ABG kernel
- identify where `gsdlc` correctly uses the model
- identify where `gsdlc` drifted or overfit implementation choices into the spec

This sequencing matters. Otherwise `gsdlc` review keeps moving while the ABG substrate is still changing under it.

## Decision Rules

These rules should be explicit in the strategy.

1. **Do not add a new primitive because one workflow wants it.**
   A new primitive should require repeated failure across multiple independent feature sets.

2. **Prefer composition law over ontology growth.**
   If `gsdlc`, `PoC`, `Discovery`, and `Research` can be modeled as compositions/macros over the same substrate, do not turn them into first-class primitives.

3. **Prefer build/runtime realization over kernel expansion.**
   If the gap is “how a worker is invoked,” “how an event store is implemented,” or “how a transport dispatches F_P,” it is not an ABG primitive gap.

4. **Prefer spec clarification over primitive growth.**
   If the model is already present but underspecified, clarify the contract before expanding the type system.

5. **Only carry 1.0-critical gaps into the kernel plan.**
   A feature set can be useful as a pressure test without being part of the 1.0 ship set.

## Likely Pressure-Test Outcomes

This is the likely shape, based on the current discussion:

- `gsdlc`, `PoC`, `Discovery`, `Research`
  likely composition/workflow macros, not new primitives

- multi-worker scheduling and consensus
  likely already pressure existing `Worker`, `Job`, `Rule`, `Consensus`, and scheduler semantics; may require stronger orchestration rules, but not obviously new ontology

- observer-driven intent formation
  likely exposes a missing formal concept such as `ObserverModel`, but this may still be expressible as a composition over `Context[]` rather than a new primitive

- dynamic routing from intent to composition set
  likely composition-law / dispatch-table territory, not kernel primitive territory

- distributed execution
  likely build/runtime backend territory, not kernel primitive territory

## Suggested ABG 1.0 Closure Criteria

Claude should propose explicit closure criteria like:

1. Every MVP-critical future feature set is representable with current GTL + ABG + composition law.
2. No unresolved missing primitive remains for `gsdlc`, `PoC`, `Discovery`, or `Research`.
3. Remaining gaps are clearly classified as:
   - spec clarification
   - build/runtime realization
   - post-1.0 expansion
4. The kernel/service boundary is stable enough that `gsdlc` can be reviewed against it.

## Important Framing

The key architectural distinction to preserve:

- `abg` is the kernel / single-hop execution substrate
- `GTL` is the SDK/IR
- `gsdlc`, `PoC`, `Discovery`, `Research` are solution compositions over that substrate

This is what keeps ABG 1.0 achievable without inventing unnecessary ontology.
